## 成功之处

### 1. PermissionHandler 架构设计精准（Story 3-1 → 3-2）

PermissionHandler 的三阶段评估流程（globalMode → persistentRules → sessionOverrides → requiresApproval）在 Story 3-1 建立后，Story 3-2 仅需添加 `deleteRule` 和 `persistGlobalMode` 方法即可完成配置管理。与 Epic 2 的 ToolRenderable 一样，这验证了"先建引擎，后建管理 UI"的 Story 拆分策略。

### 2. CheckedContinuation 桥接模式堪称典范

Story 3-1 的 `canUseTool` 回调 → `CheckedContinuation` → UI 弹窗 → `resume` 的桥接方案，完美解决了 `@Sendable` SDK 回调与 `@MainActor` UI 之间的异步鸿沟。这个模式在后续 Epic 中可以复用于任何"SDK 异步回调需要用户交互"的场景。

### 3. streamInput() 重构是关键架构决策（Story 3-3）

Story 3-3 的 Checkpoint Review 发现 `agent.stream()` 每次开新 turn 会导致旧 turn 被截断。重构为 `agent.streamInput()` + `AsyncStream<String>` 队列模式是一个架构级的正确决策。`pendingTurnCount` 跟踪机制确保了追加消息的可靠顺序处理。这个重构解决了"Agent 运行时发送追加消息"的根本技术挑战。

### 4. IMESafeTextView 解决了重要的本地化问题

Story 3-3 发现 SwiftUI 的 `onKeyPress(.return, phases: .down)` 在中文输入法 IME 组合状态下会误触发发送。使用 NSTextView 包装器 + `hasMarkedText()` 检测的方案，确保了中文用户（和所有 IME 用户）的正确体验。这是一个容易被忽视但对用户影响巨大的修复。

### 5. selectedEventId 状态提升模式清晰（Story 3-3 → 3-4）

Story 3-3 建立了 `@Binding` 状态提升模式（renamingSessionID），Story 3-4 直接复用此模式将 `selectedEventId` 从 TimelineView 提升到 WorkspaceView。模式在连续 Story 间的传递和进化非常自然。

### 6. 300 行文件限制严格执行

Story 3-4 InspectorView 达到 284 行后主动提取 `EventDetailSections.swift` 扩展文件降至 196 行。Story 3-5 PlanView 保持在 ~260 行。团队对文件行数限制的纪律性很好。

### 7. 测试增长持续健康

427 → 560 → 589 → 615 → 661，每个 Story 都增加了有意义的测试覆盖。特别是 Story 3-5 的 46 个 ATDD 测试覆盖了 PlanStep 模型、EventMapper 映射和 PlanView 渲染三个维度。

## 挑战与改进空间

### 1. 并发竞态条件是 Epic 3 的首要挑战（Story 3-3）

**描述：** Story 3-3 的 Review 发现了 3 个关键并发问题：
1. `isRunning` 竞态条件 — 并发 Task 提前设为 false
2. `streamingText` 在并发 Task 间交错
3. `currentTask` 引用被覆盖，旧 Task 不可取消

**根因：** `agent.stream()` 的每次调用都会创建新的异步流，多个并发 Task 的生命周期管理不当导致状态交错。

**改进：** 建立 `activeTaskGeneration` 模式作为异步任务管理的标准方案。每个异步任务在循环头部检查 generation，旧 generation 自动退出。

### 2. Story 3-3 是 Epic 中最复杂的 Story

**描述：** Story 3-3 同时处理了 4 组不相关功能（Sidebar 删除、重命名、追加消息、IME），并在 Checkpoint Review 中经历了两个关键重构（streamInput 架构级重构、IME 安全替换）。

**根因：** Story 拆分时低估了"Agent 运行中追加消息"和"IME 安全"的技术复杂度。

**改进：** 涉及 SDK 行为变更和输入法交互的 Story 应单独拆分，不与其他功能合并。

### 3. Story 3-2 未记录独立测试数量

**描述：** Story 3-2 的 Completion Notes 仅记录了 "560 tests pass"，未记录独立新增测试数量。这是 Epic 2 行动项 1（每个 Story 必须记录测试数量）的部分违反。

**根因：** 行动项执行不彻底。

**改进：** 行动项需要在每个 Story 开始前的模板中强制提醒，而非依赖记忆。

### 4. AgentEventType 新增 case 的影响范围

**描述：** Story 3-5 新增 `.plan` case 到 `AgentEventType` 枚举后，所有 exhaustive switch 必须处理。测试中 `testAgentEventTypeAllCases` 的硬编码 count 从 19 更新为 20。

**根因：** `CaseIterable` 枚举的测试使用硬编码 count 而非动态计算。

**改进：** CaseIterable 枚举的测试应使用 `AgentEventType.allCases.count` 替代硬编码数字，避免每次新增 case 都需手动更新测试。

### 5. Review 延迟项追踪不够系统

**描述：** Epic 3 的 Review 延迟项（Story 3-2 的 silent error swallowing、Story 3-4 的 CopyButton 放置位置）未统一记录到 deferred-work.md。

**根因：** Review 延迟项仅记录在 Story 文件中，未同步到统一追踪文件。

**改进：** 每次 Review 产生延迟项时，必须同步写入 deferred-work.md，包含处置决策。

## 关键洞察

1. **并发安全是 Agent 应用的核心挑战** — Epic 3 的 Story 3-3 暴露了 3 个并发竞态问题，全部在 Review 中捕获。这证明 Review 机制在异步代码质量保障中不可替代。`activeTaskGeneration` 模式应成为项目中所有异步任务管理的标准方案。

2. **SDK 行为变更需要架构级验证** — Story 3-3 从 `agent.stream()` 重构为 `agent.streamInput()` 的经历说明，当 Story 涉及 SDK 核心调用方式变更时，需要在实现前进行更深入的架构验证，而非在 Checkpoint Review 中才发现根本性设计问题。

3. **IME 安全是 macOS 中文用户的刚需** — `IMESafeTextView` 的引入解决了一个容易被忽视但对中文（以及日文、韩文等）用户影响巨大的问题。任何涉及键盘输入的 Story 都应包含 IME 安全审查。

4. **事件类型枚举是项目的"脊柱"** — `AgentEventType` 新增 `.plan` case 后编译器强制所有 exhaustive switch 更新，这是 Swift 类型安全的巨大优势。新增事件类型时只需在 3 处添加 case（TimelineView、InspectorView、colorForEventType），编译器保证不会遗漏。

5. **Story 顺序仍有优化空间** — Story 3-3 同时包含 4 组功能，导致复杂度集中。如果将"追加消息 + IME"拆分为独立 Story，与"Sidebar 删除/重命名"分离，复杂度会更均匀。

6. **Epic 2→3 准备任务全部就绪** — Epic 2 回顾中标记的 4 项 Epic 3 准备任务全部完成，验证了"回顾驱动的准备"机制的有效性。
